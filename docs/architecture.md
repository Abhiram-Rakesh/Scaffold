# Architecture

## Overview

Scaffold implements a **hub-and-spoke model** for Terraform state management and CI/CD across multiple AWS accounts.

## Components

### Centralized State Backend (Platform Account)

| Resource | Purpose |
|----------|---------|
| S3 Bucket | Stores Terraform state files, one per environment |
| DynamoDB Table | Provides distributed locking to prevent concurrent applies |
| KMS Key | Encrypts state files at rest; enables cross-account access |

### Per-Environment Resources (Spoke Accounts)

| Resource | Purpose |
|----------|---------|
| IAM OIDC Provider | Trusts GitHub's OIDC issuer (one per account) |
| IAM Role | Assumed by GitHub Actions via OIDC; holds Terraform permissions |

### Repository Files (Generated by Scaffold)

| File | Purpose |
|------|---------|
| `.scaffold/config.json` | Tracks backend and environment configuration |
| `.github/workflows/terraform-<env>.yaml` | GitHub Actions workflow per environment |
| `<watch-dir>/providers.tf` | Terraform backend configuration |

## Data Flow

```
git push → GitHub Actions trigger
           │
           ├─ OIDC token request to GitHub's IdP
           │
           └─ sts:AssumeRoleWithWebIdentity
                         │
                         ▼
              IAM Role in Spoke Account
              (verified via OIDC trust policy)
                         │
           ┌─────────────┼─────────────┐
           │             │             │
           ▼             ▼             ▼
      S3 GetObject  DynamoDB PutItem  KMS Decrypt
      (read state)  (acquire lock)  (decrypt state)
           │
           ▼
      terraform plan/apply
           │
           ▼
      S3 PutObject + DynamoDB DeleteItem
      (save state + release lock)
```

## Security Design

### No Long-Lived Credentials
GitHub Actions never receives AWS access keys. It uses OIDC to get short-lived tokens that expire after the workflow run.

### Least Privilege
- GitHub Actions role has broad infrastructure access but cannot manage IAM or Organizations
- State backend access is scoped to the specific environment's state file path
- KMS access is limited to the specific encryption key

### State Encryption
All Terraform state is encrypted at rest using AWS KMS with automatic key rotation. Cross-account access uses KMS key policies rather than bucket ACLs.

### SCP Compatibility
Scaffold uses inline IAM policies (`aws_iam_role_policy`) instead of managed policy attachments (`aws_iam_role_policy_attachment`). This is critical in AWS Organizations environments with restrictive SCPs.
